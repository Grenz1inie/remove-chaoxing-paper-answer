## 🏗️ 技术架构

### 架构模式
- **模块化设计**：功能按职责拆分为独立模块
- **依赖注入**：通过 `@require` 声明模块依赖
- **单一职责**：每个模块只负责特定功能
- **配置分离**：配置与逻辑代码分离

### 模块划分

```
┌─────────────────────────────────────┐
│         main.user.js                │
│         (主入口协调)                 │
└────────────┬────────────────────────┘
             │
    ┌────────┼────────┬─────────┐
    │        │        │         │
    ▼        ▼        ▼         ▼
┌────────┐ ┌────┐ ┌────────┐ ┌──────────┐
│ Config │ │ DOM│ │   UI   │ │ Button   │
│ 配置   │ │操作│ │ 界面   │ │ 控制器   │
└────────┘ └────┘ └────────┘ └──────────┘
```

### 数据流向

```
用户操作
   ↓
按钮事件 (button-controller.js)
   ↓
DOM操作 (dom-handler.js)
   ↓
UI更新 (ui.js)
   ↓
视觉反馈
```

## 🔧 技术特性

### 1. 模块化
- **优点**：代码结构清晰，易于维护和扩展
- **实现**：通过 Tampermonkey 的 `@require` 实现模块加载
- **依赖管理**：明确的依赖关系，避免循环依赖

### 2. 配置化
- **优点**：修改配置无需改动业务逻辑
- **实现**：所有配置集中在 `config.js`
- **扩展性**：易于添加新的配置项

### 3. 封装性
- **优点**：模块之间低耦合，高内聚
- **实现**：每个模块通过 `window` 对象导出接口
- **维护性**：修改内部实现不影响其他模块

### 4. 可读性
- **优点**：代码易读易懂，降低学习成本
- **实现**：清晰的命名、详细的注释、JSDoc文档
- **可维护**：新人也能快速上手

## 🎨 设计原则

### 1. KISS（Keep It Simple, Stupid）
- 保持代码简单直接
- 避免过度设计
- 功能实现清晰明了

### 2. DRY（Don't Repeat Yourself）
- 避免代码重复
- 提取公共逻辑
- 复用现有模块

### 3. 单一职责原则
- 每个模块只做一件事
- 职责明确，边界清晰
- 便于测试和维护

### 4. 开放封闭原则
- 对扩展开放
- 对修改封闭
- 通过配置实现定制
